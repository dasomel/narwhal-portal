# ADR: Cost Insights 비용 산정 기준 — 온프레미스 실원가 방식

- **Status**: Draft (검토 대기)
- **Date**: 2026-06-12
- **관련 문서**: [service-visibility-suite-design §4.4](./superpowers/specs/2026-05-13-service-visibility-suite-design.md), `src/lib/cost.ts`, [ROADMAP — 비용 가시성](./ROADMAP.md)

## 1. Context

Cost Insights는 현재 `사용량 × 단가` 모델로 비용을 추정하지만, 단가 기본값
(`COST_CPU_HOURLY=0.04`, `COST_MEM_GB_HOURLY=0.005`, `COST_STORAGE_GB_HOURLY=0.0001`)은
**산출 근거가 없는 placeholder**다. 수치는 퍼블릭 클라우드 온디맨드 요율 수준의 추정치로,
narwhal 클러스터(온프레미스 VM)의 실제 원가와 무관하다. 통화 단위, 측정 기준(usage vs
request), 유휴 비용 처리, 월 환산 계수도 정의되어 있지 않다.

이 ADR은 **온프레미스 실원가(TCO 기반) 방식**으로 단가 산출 기준을 정의한다.

## 2. Decision

### 2.1 목적: Showback (Chargeback 아님)

비용 표시는 **인식·비교 목적(showback)** 이다. 실제 청구·정산(chargeback)이 아니므로
정밀도보다 **일관성과 재현 가능한 산출 근거**를 우선한다.

### 2.2 원가 구성요소 (TCO)

월 총원가는 다음 항목의 합으로 정의한다:

| 항목 | 정의 | v1 포함 여부 |
|------|------|:---:|
| 하드웨어 감가상각 | 호스트 머신 구매가 ÷ 내용연수(기본 **36개월**, 정액법) | ✅ |
| 전력 | 평균 소비전력(kW) × 730h × 전기요금(₩/kWh) | ✅ |
| 스토리지 감가상각 | 디스크(SSD) 구매가 ÷ 내용연수(36개월) | ✅ |
| 상면·회선 | 코로케이션/네트워크 회선 비용 | ❌ (자택/사무실 호스트 — 해당 없음) |
| 운영 인건비 | 클러스터 운영 공수 | ❌ (v1 제외 — 산정 기준 합의 후 v2 검토) |

### 2.3 클러스터 할당 비율 (β)

클러스터가 전용 머신이 아닌 **공유 호스트의 VM**으로 운영되므로, 호스트 원가 중
클러스터 귀속분을 할당 비율 β로 배분한다:

```
β = max( 클러스터 vCPU 합 / 호스트 코어 수,
         클러스터 RAM 합  / 호스트 RAM )
```

> max를 쓰는 이유: 병목 자원이 호스트 증설(=비용)을 결정하기 때문.
> 전용 머신이면 β = 1.0.

### 2.4 자원별 단가 산출 공식

```
C_cluster (월)  = (호스트 감가상각 + 전력비) × β
C_hourly        = C_cluster / 730

CPU 단가   (₩/vCPU-hr) = C_hourly × W_cpu / 클러스터 vCPU 합
MEM 단가   (₩/GB-hr)   = C_hourly × W_mem / 클러스터 RAM(GB) 합
STORAGE 단가 (₩/GB-hr) = (디스크 월 감가상각 / 디스크 총 용량 GB) / 730
```

- **W_cpu : W_mem = 0.65 : 0.35** (기본값). 서버 원가에서 CPU가 차지하는 비중이
  크다는 업계 관행(OpenCost custom pricing 분해 방식)을 따른 가중치이며, 합은 항상 1.
- 스토리지는 컴퓨트와 분리해 디스크 원가에서 직접 산출한다 (프로비저닝 용량 기준).

### 2.5 측정·배분 기준

| 항목 | 기준 | 비고 |
|------|------|------|
| 과금 기준량 | **실사용량(usage)** — `rate(container_cpu_usage_seconds_total[1h])`, `working_set_bytes`, PVC `used_bytes` | v1 유지. FinOps 표준인 `max(usage, request)` 전환은 후속 결정 항목 (§5) |
| 유휴(idle) 비용 | 서비스에 **배분하지 않음** | 클러스터 스코프 합계와 서비스 합계의 차이로 자연 표현됨. UI에 "서비스 합계 ≠ 클러스터 총원가" 주석 필요 |
| 시스템 네임스페이스 | 일반 네임스페이스와 동일하게 표시 | 별도 "platform overhead" 분류는 v2 검토 |
| 월 환산 계수 | **730h** (= 365일 × 24h ÷ 12) | 현재 코드는 720 사용 — 730으로 통일 (§4 후속 변경) |
| 통화 | **KRW** | 단가 env 값을 ₩ 기준으로 입력. 통화 라벨용 `COST_CURRENCY` env 신설 (§4) |

### 2.6 재산정 주기

- **연 1회** 정기 재산정, 또는 하드웨어 교체 / 전기요금 변경 / 클러스터 토폴로지 변경 시 수시.
- 재산정 시 본 문서 부록(§6)에 이력을 기록하고 `.env` 값을 갱신한다.

## 3. 예시 계산 — 현재 narwhal 토폴로지

> ⚠️ 아래 굵은 입력값은 **예시(placeholder)** 다. 실제 구매가·소비전력·전기요금으로
> 치환한 뒤 확정한다 (§5 미결정 항목).

**입력 파라미터:**

| 파라미터 | 예시값 | 근거 |
|---------|-------:|------|
| 호스트 구매가 | **₩4,000,000** | [TBD: 실제 구매가] |
| 호스트 사양 | **16코어 / 64GB** | [TBD: 실제 사양] |
| 호스트 평균 소비전력 | **0.12 kW** | [TBD: 실측 또는 TDP 추정] |
| 전기요금 | **₩130/kWh** | [TBD: 실제 요금제] |
| 디스크 | **2TB SSD, ₩200,000** | [TBD] |
| 클러스터 할당 | 12 vCPU / 36GB / 180GB | VM 6대 × (2vCPU, 6GB, 30GB) — README 토폴로지 |

**산출:**

```
감가상각  = 4,000,000 / 36         = ₩111,111/월
전력비    = 0.12 × 730 × 130       = ₩ 11,388/월
β         = max(12/16, 36/64)      = 0.75
C_cluster = (111,111 + 11,388) × 0.75 = ₩91,874/월
C_hourly  = 91,874 / 730           = ₩125.9/시간

CPU 단가  = 125.9 × 0.65 / 12      = ₩6.82 /vCPU-hr
MEM 단가  = 125.9 × 0.35 / 36      = ₩1.22 /GB-hr
STG 단가  = (200,000/36) / 2048 / 730 = ₩0.0037 /GB-hr
```

**`.env` 반영:**

```bash
COST_CPU_HOURLY=6.82
COST_MEM_GB_HOURLY=1.22
COST_STORAGE_GB_HOURLY=0.0037
COST_CURRENCY=KRW        # 신설 (§4)
```

## 4. Consequences — 후속 변경 항목

| # | 변경 | 위치 | 규모 |
|---|------|------|------|
| 1 | 월 환산 720 → **730** 통일 | `src/lib/cost.ts` (`totalMonthly` 주석·계산), spec §4.4 | 소 |
| 2 | `COST_CURRENCY` env 추가 + UI 통화 라벨 (현재 $ 하드코딩 여부 확인) | `cost.ts`, `/cost` 페이지, catalog cost 탭 | 소 |
| 3 | UI 주석: "유휴 자원 비용은 서비스에 배분되지 않음 (showback 기준)" | `/cost` 글로벌 화면 | 소 |
| 4 | 단가 기본값(코드 내 0.04 등)은 유지하되 "근거 없는 fallback"임을 주석 명시, 운영 환경은 `.env` 필수 | `cost.ts:13-26` | 소 |
| 5 | spec §4.4에 본 ADR 링크 추가 | design spec | 소 |

## 5. 미결정 항목 (확정 필요)

1. **실제 입력값**: 호스트 구매가, 사양, 소비전력, 전기요금, 디스크 구매가 (§3 [TBD] 치환).
2. **과금 기준량 전환**: usage → `max(usage, request)` (예약했지만 안 쓰는 자원에 비용 부과).
   request 메트릭(`kube_pod_container_resource_requests`) PromQL 추가 필요 — v1.1 후보.
3. **W_cpu:W_mem 가중치** 0.65:0.35 적정성 — 호스트가 메모리 비중 높은 구성이면 조정.
4. **운영 인건비** 포함 여부와 산정 방식 (v2).
